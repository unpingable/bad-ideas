# PM-Claude / Project-Manager Claude — imperial coordinator as default response to differentiation

**Receipt:** Surfaced 2026-05-18 with the operator opener *"I swear I should just have a project-manager claude instance"* — the temptation moved through the operator before the meta-check caught it. Chatty's pre-processing diagnosed *"good idea, dangerous implementation"* and named the failure mode (PM-as-middle-manager / *"haunted Jira with better prose"*). The historical receipts here are unusually strong: AG (agent_gov) actually occupied the PM-cortex role legitimately when the constellation was tightly coupled. The temptation now is to keep wearing that hat after the topology has changed.

This is the lane's first entry about *AI orchestration shape* rather than product/protocol/social-substrate temptation. The decoy isn't a thing being built in the world; it's a role being delegated inside the operator's own workflow.

## Seductive shape

A dedicated Claude instance ("PM-Claude," "Project Manager Claude") that coordinates across the operator's growing repo constellation. Tracks open gaps, debt, branch state, "waiting on X," which repo owns which seam. Produces pickup packets, prioritization lists, dependency maps. Updates continuity notes, gap indices, `CLAUDE.md` files. The pitch is *"air traffic control for the constellation"* — sounds modest and useful.

## Real pressure

The diagnosis is correct. The operator's constellation has differentiated: RPP, NQ, WLP, Wicket, Continuity, Nightshift each acquired local sovereignty. The cost of orientation across repos has gone up. The cognitive load of remembering which seam is half-open in which repo is real. AG used to absorb this cost; once differentiation occurred, AG's absorption became a category error and the orientation cost reappeared at the operator level.

## Why building it as PM-Claude is wrong

1. **Category name recruits category ambition.** *"Project Manager Claude"* names a role after a category, not a function. Category-named roles drift toward the full category job description. A PM that started as air traffic control accretes ticket generation, stakeholder-alignment vocabulary, prioritization authority, ownership claims — *"haunted Jira instance with better prose,"* per chatty.

2. **Imperial-AG revival.** AG wore the PM-cortex hat legitimately when the constellation was tightly coupled. A new PM-Claude would re-import that authority surface after the topology has changed: project-specific doctrine would leak sideways through the planning surface, RPP/NQ/WLP local sovereignty would erode, and *"everything routes through PM"* would re-emerge.

3. **Sovereign creep by utility.** A useful coordinator is reached for; a reached-for coordinator accumulates default authority; default authority calcifies into command authority. Utility is the failure path, not the bug — the bug is allowing utility to mint authority.

4. **The temptation pre-empts its own check.** *"I should just have a PM-Claude"* — the move from felt-pressure to delegate-the-pressure happens in one operator sentence. That speed is the tell. Differentiating the constellation took multiple passes; un-differentiating it back into a single planning cortex should not be a one-sentence move.

## Useful isotope

The portable extract is **not** PM-Claude. It is the **Constellation Clerk** shape, filed separately at `gnat/good_actually/constellation-clerk.md` as a *candidate* coordination surface under the non-sovereign perimeter doctrine. Status: candidate; build authorization: none.

Core keeper: *the clerk describes constellation state; it does not decide constellation direction.* Sharper: *the clerk may prepare pickup packets; it may not mint priorities.* The clerk is what's needed *after* differentiation — substrate-neutral, advisory, ignorable, never where doctrine lives.

Also extracted: a doctrine extension to `gnat/good_actually/non-sovereign-perimeter-doctrine.md` naming **internal constellation coordination** as a second application domain of the same doctrine that already governs external perimeter lenses. The four sub-rules transpose cleanly (lens advises / receipted interpretation / advisory+time-limited / subscribe-and-swap / pressure-shape vocabulary → state-tracking vs decision-making, snapshot vs authoritative, ignorable pickup packets, state-shape vocabulary).

## Not authorized as

- a dedicated "PM-Claude" or "Project Manager Claude" instance
- a successor to AG's old coordination role
- a prioritization authority across repos
- a "single source of truth" for project state
- an entity that owns gaps, seams, or boundaries
- excuse to deprioritize ship-code-and-ship-book

Permitted as: the Constellation Clerk candidate (filed separately), pending ratification with explicit authority constraints.

## Forbidden unless

Promotion from "candidate clerk" to "active coordination surface" requires all of:

1. The clerk's *can-say* and *cannot-say* lists have been ratified — including which vocabulary is forbidden (no *should*, no *next*, no *single source of truth*, no *owns this*).
2. Each participating project has explicitly subscribed; clerk output is ignorable by default.
3. The clerk's authorization is time-limited with a stated expiration; renewal requires re-ratification.
4. An audit pathway exists for catching sovereign-creep behavior (a clerk that drifts into prescription gets demoted to silence).
5. Ship-code and ship-book obligations are not the active critical path.

Until all five hold, the answer is *"keep the orientation cost where it lives — in the operator — and accept that differentiation has cognitive load."*

## Keeper line

> Coordination may reduce orientation cost, but it must not acquire command authority by being useful. Name the function (clerk), not the category (manager); the useful project manager is the one least allowed to manage.
